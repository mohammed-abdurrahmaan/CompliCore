Token Technical Details:
- Algorithm: HS256 (HMAC-SHA256)
- Signing secret: JWT_SECRET env var (defaults to 'complicore-secret-key-2025' if not set)
- Expiry: datetime.now(UTC).timestamp() + 86400 * 7 (7 days from creation)
- Payload: {user_id: UUID, role: string, exp: float}
- No refresh token. After 7 days, user must log in again.
- Token is NOT revoked on logout (client-side only). A copied token remains valid until expiry.

Unknown Route Behavior

No 404 handler: There is no catch-all route defined. If an authenticated user navigates to an unknown path like /settings or /unknown, the DashboardLayout renders with an empty content area (no error page). If unauthenticated, the ProtectedRoute redirects to /login.

12 Known Gaps & Missing Features

Features that do NOT exist in the current implementation

  • No "Forgot Password" flow - No password reset mechanism. If a user forgets their password, there is no recovery path.
  • No email verification - Registration is instant. Email is not verified. Users can register with non-existent emails.
  • No password strength enforcement - Placeholder says "Min 6 characters" but passwords of any length (even 1 character) are accepted by both frontend and backend.
  • No password complexity rules - No requirements for uppercase, lowercase, numbers, or special characters.
  • No rate limiting on auth endpoints - Login and register endpoints have no rate limiting. Vulnerable to brute-force attacks.
  • No account lockout - No lockout after repeated failed login attempts.
  • No server-side token revocation - Logout only clears client state. Token remains valid server-side until expiry.
  • No refresh token - After 7 days, session expires with no way to extend without re-entering credentials.
  • No 404 page - Unknown routes show blank content area instead of a helpful error page.
  • No role validation on backend - The role field accepts any string via API (e.g., "admin", "superuser"). The UI dropdown constrains to employer/actuary.
  • No email case normalization - "User@Test.com" and "user@test.com" are treated as different accounts.
  • No session activity tracking - No last login timestamp or session activity log.
  • Login form has no frontend validation - Empty submissions go directly to the API.
